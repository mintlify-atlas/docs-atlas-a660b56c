The `github/docs` repository is a monorepo containing both the documentation content and the application that serves it. Understanding the directory structure helps you find the right place to make changes, whether you're editing an article or working on the platform itself.

## Content directory

The `/content` directory holds all English Markdown documentation. It is organized by product, matching the URL structure of docs.github.com:

```
content/
├── actions/          → docs.github.com/en/actions
├── codespaces/       → docs.github.com/en/codespaces
├── github-cli/       → docs.github.com/en/github-cli
├── rest/             → docs.github.com/en/rest
├── admin/            → docs.github.com/en/admin
└── ...
```

Every directory has an `index.md` that acts as the landing page for that product or category. The `index.md` must include a `children` frontmatter property listing its sub-pages — the site only renders paths listed there. Pages not referenced in `children` return a 404.

Content files use:

- **YAML frontmatter** — metadata like `title`, `intro`, `versions`, `redirect_from`, and layout options
- **Markdown** — standard CommonMark with GitHub Flavored Markdown extensions
- **Liquid templating** — conditional versioning, reusables, variables, and tool-switching tags

<Tip>
  The `/content` directory is where external contributors are welcome. Edits to `.md` files here — fixing typos, improving explanations, updating outdated content — are the most common type of contribution.
</Tip>

## Data directory

The `/data` directory contains structured content that is referenced from Markdown files via Liquid tags. It is not directly served as pages.

| Subdirectory | Purpose |
|--------------|---------|
| `data/reusables/` | Short Markdown snippets reused across multiple articles. Referenced with `{% data reusables.topic.reusable-name %}` |
| `data/variables/` | Short text strings (product names, version numbers). Referenced with `{% data variables.product.product_name %}` |
| `data/learning-tracks/` | YAML definitions for learning track sequences shown on product landing pages |
| `data/graphql/` | GraphQL schema data used to auto-generate the GraphQL reference docs |
| `data/features/` | Feature flag definitions used to gate content by product version |

<Note>
  External contributors may edit files in `data/reusables/` and `data/variables/`. Other subdirectories under `/data` are typically managed by automated pipelines or GitHub staff.
</Note>

## Source directory

The `/src` directory contains the Node.js application that serves docs.github.com. It follows a **subject folder** pattern — each subdirectory represents a distinct capability of the site, with its own `README.md`, `lib/`, `middleware/`, `pages/`, `components/`, and `tests/`.

## The server

The docs.github.com site is a **dynamic Node.js web server** built on Express with Next.js for React rendering. It was not always this way:

```
Ruby on Rails → Jekyll (static) → Nanoc (static) → Node.js/Express/Next.js (current)
```

Despite the stack changes, many authoring conventions from the original Jekyll era are preserved:

- Content is written in Markdown files in `/content`
- Files support [Liquid templating](https://shopify.github.io/liquid/)
- Data files in `/data` are available to templates via `{% data %}` tags
- Markdown files use Jekyll-style frontmatter
- The `redirect_from` frontmatter behavior (from the `jekyll-redirect-from` plugin) is still supported

The server handles:

- **HTTP redirects** — including `redirect_from` frontmatter and external site redirects
- **Language detection** — via `Accept-Language` headers and URL prefixes (`/en/`, `/ja/`, etc.)
- **Version routing** — URLs include the product version for GitHub Enterprise Server (e.g., `/en/enterprise-server@3.12/admin/...`)
- **Dynamic content** — Liquid conditionals and versioning are evaluated at request time

## Contributing guide directory

The `/contributing` directory contains guides for working on the repository. Most of its content has been superseded by the official [contributing documentation](https://docs.github.com/en/contributing) on docs.github.com, but some files (like `development.md` and `liquid-helpers.md`) remain useful references for local setup and authoring conventions.
